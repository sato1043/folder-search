# 手動E2Eテスト項目リスト

本ドキュメントは、自動化が困難またはコスト対効果が低いE2Eテスト項目を管理する。
新機能の追加時に自動化を検討し、自動化できないものはここに追記する。

## 記載フォーマット

```
### MET-XXX: テスト名

- **Phase**: PhaseN
- **前提条件**: テスト実施前に必要な状態
- **手順**:
  1. 操作1
  2. 操作2
- **期待結果**: 確認する内容
- **備考**: 補足事項
```

## テスト項目

### MET-001: embeddingモデルの起動時自動ダウンロード

- **Phase**: Phase 3
- **前提条件**: embeddingモデルが未ダウンロード状態（`rm -f src-tauri/target/debug/models/model.onnx src-tauri/target/debug/models/tokenizer.json`）
- **手順**:
  1. `pnpm tauri dev` でアプリを起動する
  2. ローディングオーバーレイに「Embeddingモデルをダウンロード中...」と進捗率が表示されることを確認
  3. ダウンロード完了後、オーバーレイが消えてアプリが操作可能になることを確認
  4. 設定ダイアログを開き、一般セクションのベクトル検索モデルが「ダウンロード済み」と表示されることを確認
- **期待結果**: model.onnx（約450MB）とtokenizer.jsonが自動ダウンロードされ、モデルが利用可能になる。LLM自動ロードと並列実行される場合はオーバーレイに両方の状態が表示される
- **備考**: ネットワーク環境に依存。モデル削除手順は `.claude/CLAUDE.md` のコマンド一覧を参照

### MET-001a: 設定ダイアログからのembeddingモデル手動ダウンロード

- **Phase**: Phase 5
- **前提条件**: embeddingモデルが未ダウンロード状態
- **手順**:
  1. アプリ起動後、自動ダウンロードをネットワーク切断等で失敗させる（または自動DL完了前に設定ダイアログを開く）
  2. 設定ダイアログの一般セクションでベクトル検索モデルが「未ダウンロード」と表示されることを確認
  3. ダウンロードボタン（↓）をクリック
  4. ダウンロード進捗が表示されることを確認
  5. 完了後「ダウンロード済み」に変わることを確認
- **期待結果**: 設定ダイアログからの手動ダウンロードでモデルが取得できる
- **備考**: 自動ダウンロード失敗時のリカバリ手段

### MET-002: ベクトルインデックスの構築と進捗表示

- **Phase**: Phase 3
- **前提条件**: フォルダ選択済み、全文検索インデックス構築済み、embeddingモデルダウンロード済み
- **手順**:
  1. サイドバーの「ベクトルインデックス構築」ボタンをクリック
  2. チャンク処理の進捗が表示されることを確認
  3. 完了後「N チャンク登録済み」と表示されることを確認
- **期待結果**: ファイルがチャンク分割され、embeddingが生成され、HNSWインデックスが構築される
- **備考**: 1000ファイル規模では数分かかる可能性がある

### MET-003: ハイブリッド検索の実行

- **Phase**: Phase 3
- **前提条件**: 全文検索インデックスとベクトルインデックスの両方が構築済み
- **手順**:
  1. 検索モードを「ハイブリッド検索」に切り替え
  2. 検索クエリを入力してEnter
  3. 検索結果が表示されることを確認
  4. 結果をクリックしてプレビューが表示されることを確認
- **期待結果**: 全文検索とベクトル検索の結果がRRFで統合された結果が表示される
- **備考**: 全文検索のみの結果と比較して、意味的に関連するドキュメントが上位に来ることを確認する

### MET-004: 小規模フォルダではダイアログなしでインデックス作成される

- **Phase**: Phase 6
- **前提条件**: 500ファイル未満の .txt/.md を含むフォルダを用意する
- **手順**:
  1. 「フォルダを選択」ボタンをクリック
  2. 小規模フォルダを選択する
- **期待結果**: 確認ダイアログが表示されず、即座にインデックス作成が開始・完了する
- **備考**: なし

### MET-005: 大規模フォルダで確認ダイアログが表示される

- **Phase**: Phase 6
- **前提条件**: 500ファイル以上の .txt/.md を含むフォルダを用意する
- **手順**:
  1. 「フォルダを選択」ボタンをクリック
  2. 大規模フォルダを選択する
  3. 確認ダイアログが表示されることを確認する
  4. ファイル数・合計サイズ・警告メッセージが表示されることを確認する
- **期待結果**: 閾値条件に応じた警告メッセージを含む確認ダイアログが表示される
- **備考**: 閾値: ファイル数>=500, 合計サイズ>=100MB, 推定チャンク>=40,000, 最大ファイル>=10MB

### MET-006: 確認ダイアログから進捗表示への遷移

- **Phase**: Phase 6
- **前提条件**: MET-005で確認ダイアログが表示された状態
- **手順**:
  1. 確認ダイアログの「開始」ボタンをクリック
  2. 全文検索インデックス構築の進捗バーが表示されることを確認する
  3. 全文検索完了後、ベクトルインデックス構築の進捗バーに遷移することを確認する（embeddingモデルDL済みの場合）
  4. 完了後「インデックス作成完了」画面が表示されることを確認する
- **期待結果**: 確認→Phase1進捗→Phase2進捗→完了の順にダイアログが遷移する。各フェーズでファイル数/チャンク数とパーセンテージが更新される
- **備考**: embeddingモデル未DLの場合はPhase2はスキップされる

### MET-007: Phase1（全文検索インデックス構築）中に中断する

- **Phase**: Phase 6
- **前提条件**: MET-006の手順1を実行し、全文検索インデックス構築中の状態
- **手順**:
  1. 進捗ダイアログの「中断」ボタンをクリック
  2. 「インデックス作成を中断した」画面が表示されることを確認する
  3. OKをクリックしてダイアログを閉じる
- **期待結果**: インデックスが破棄される。サイドバーにファイル数が表示されない。全文検索・ベクトル検索ともに利用不可
- **備考**: インデックスディレクトリが削除されることを確認する

### MET-008: Phase2（ベクトルインデックス構築）中に中断する

- **Phase**: Phase 6
- **前提条件**: MET-006の手順1〜2を経て、ベクトルインデックス構築中の状態
- **手順**:
  1. 進捗ダイアログの「中断」ボタンをクリック
  2. 「インデックス作成を中断した」画面が表示されることを確認する
  3. 全文検索インデックスは「完了済み」、ベクトルインデックスは「中断」と表示されることを確認する
  4. OKをクリックしてダイアログを閉じる
  5. 全文検索が利用可能であることを確認する
- **期待結果**: 全文検索インデックスは有効のまま動作する。ベクトルインデックスは未構築。途中までのembeddingがキャッシュに保存される
- **備考**: 「全文検索のみで動作する」旨が表示される

### MET-009: Phase2中断後に同じフォルダを再選択して途中から再開する

- **Phase**: Phase 6
- **前提条件**: MET-008を実施済み（Phase2中断済み、途中キャッシュ保存済み）
- **手順**:
  1. 同じフォルダを再選択する
  2. インデックス作成が完了するまで待つ
  3. ベクトルインデックスのチャンク数がフル構築時と同じになることを確認する
- **期待結果**: 差分更新パスにより、前回中断時に保存済みのembeddingが再利用され、未処理ファイルのみ新規にembedding生成される。フル構築より短時間で完了する
- **備考**: 所要時間を比較するために、フル構築時の時間を事前に計測しておくとよい

### MET-010: 破損インデックスがフォルダ選択時に検出・削除される

- **Phase**: Phase 6
- **前提条件**: 一度フォルダを選択してインデックスを構築済み
- **手順**:
  1. アプリを終了する
  2. `{appDataDir}/index/{hash}/fulltext/` 内のファイルを破損させる（例: `meta.json` の内容を書き換え）
  3. アプリを再起動する
  4. 同じフォルダを再選択する
  5. エラーなくインデックスが再構築されることを確認する
- **期待結果**: 破損インデックスがフォルダ選択時に検出・自動削除され、新しいインデックスが正常に構築される
- **備考**: ベクトルキャッシュの破損テストも同様に実施可能（`{appDataDir}/index/{hash}/vector/manifest.json` を破損させる）
