# 要件定義書

## 1. プロジェクト概要

### 1.1 プロジェクト名

Folder Search

### 1.2 目的

ローカルに蓄積されたナレッジベース（プレーンテキスト・マークダウンファイル群）から、すばやく正確に知識を引き出すデスクトップアプリケーションを提供する。

### 1.3 背景

- ナレッジが大量のローカルファイルに分散しており、必要な知識を見つけるのに時間がかかる
- 既存のファイル検索ツール（OS標準検索、grep等）ではキーワード一致に限定され、意味的な検索ができない
- クラウドサービスに依存せず、完全ローカルで動作するソリューションが求められる

### 1.4 ターゲットユーザー

- 大量のテキストファイル・マークダウンファイルをナレッジベースとして運用している個人
- プライバシーやセキュリティの観点からクラウドサービスを利用できない・したくないユーザー

## 2. 機能要件

### 2.1 フォルダ管理

- FR-001: ナレッジベースとなるフォルダを指定できる
- FR-002: 複数フォルダの登録に対応する
- FR-003: フォルダ内のファイルを再帰的に走査する
- FR-004: 対象ファイル形式はプレーンテキスト（.txt）とマークダウン（.md）を最低限サポートする

### 2.2 全文検索

- FR-010: キーワードによる全文検索ができる
- FR-011: 日本語の形態素解析に対応する（lindera + IPAdic）
- FR-012: 検索結果を関連度順にランキング表示する
- FR-013: 検索結果から原文をプレビュー表示できる
- FR-014: 検索結果から原文ファイルを開くことができる
- FR-015: ファイルの変更を検知し、インデックスを差分更新する

### 2.3 ベクトル検索

- FR-020: 意味的な類似性に基づく検索ができる
- FR-021: ファイルをチャンクに分割してembeddingを生成する
- FR-022: ハイブリッド検索（全文検索 + ベクトル検索のスコア統合）を提供する
- FR-023: embeddingモデルはローカルで動作する（ONNX Runtime）

### 2.4 ローカルLLM

- FR-030: ローカルLLMによる自然言語での質問応答（RAG）を提供する
- FR-031: llama.cppを単一バイナリに統合する
- FR-032: GPU VRAMサイズに応じてモデルを選択・切り替えできる
- FR-033: モデルをHuggingFace等から動的にダウンロードできる
- FR-034: LLMの回答に参照元ファイル名・該当箇所を付記する
- FR-035: 参照元の原文をワンクリックで表示できる

### 2.5 インデックス管理

- FR-040: ファイル監視による自動インデックス更新
- FR-041: インデックスの手動再構築
- FR-042: インデックス状態の表示（件数、最終更新日時、サイズ）

## 3. 非機能要件

### 3.1 検索応答性（最重要）

- NFR-001: 全文検索の応答時間は1000ファイル規模で100ms以内を目標とする
- NFR-002: ベクトル検索の応答時間は1000ファイル規模（数万チャンク）で500ms以内を目標とする
- NFR-003: ハイブリッド検索の応答時間は1秒以内を目標とする
- NFR-004: LLM回答生成はストリーミング表示とし、最初のトークンが3秒以内に表示されることを目標とする
- NFR-005: 検索精度について、関連文書が上位10件以内に含まれる割合（Recall@10）を80%以上とする

### 3.2 プラットフォーム互換性

- NFR-010: Windows 10以降で動作する
- NFR-011: macOS 12 (Monterey) 以降で動作する
- NFR-012: ひとつのコードベースで両プラットフォームをサポートする

### 3.3 ローカル完結性

- NFR-020: 検索・回答生成は完全にローカルで動作する（ネットワーク不要）
- NFR-021: モデルダウンロード時のみネットワーク接続を要する

### 3.4 リソース効率

- NFR-030: アイドル時のCPU使用率は5%以内とする
- NFR-031: アイドル時のメモリ使用量は500MB以内とする
- NFR-032: インデックスサイズは元ファイル総サイズの2倍以内を目標とする

### 3.5 保守性

- NFR-040: TDD（テスト駆動開発）で開発する
- NFR-041: E2Eテストを自動化する（自動化不可の場合は手動テスト項目として管理する）
- NFR-042: ドキュメントを設計変更に追従して更新する

## 4. 技術制約

- TC-001: Tauri v2をアプリケーションフレームワークとして使用する
- TC-002: フロントエンドはReact + TypeScriptで実装する
- TC-003: ネイティブレイヤーはRustで実装する
- TC-004: 全文検索エンジンはtantivyを使用する
- TC-005: 日本語形態素解析はlinderaを使用する
- TC-006: ローカルLLMはllama.cppを使用し、単一バイナリに統合する
- TC-007: LLMモデルはGGUF形式を使用する

## 5. 想定規模

| 項目 | 想定値 |
|---|---|
| ナレッジベースのファイル数 | 約1,000ファイル |
| ファイルあたり平均サイズ | 5-50KB |
| ナレッジベース総サイズ | 5-50MB |
| チャンク数（ベクトル検索用） | 5,000-50,000 |
