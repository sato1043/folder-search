# 要件定義書

## 1. プロジェクト概要

### 1.1 プロジェクト名

Folder Search

### 1.2 目的

ローカルに蓄積されたナレッジベース（プレーンテキスト・マークダウンファイル群）から、すばやく正確に知識を引き出すデスクトップアプリケーションを提供する。

### 1.3 背景

- ナレッジが大量のローカルファイルに分散しており、必要な知識を見つけるのに時間がかかる
- 既存のファイル検索ツール（OS標準検索、grep等）ではキーワード一致に限定され、意味的な検索ができない
- クラウドサービスに依存せず、完全ローカルで動作するソリューションが求められる

### 1.4 ターゲットユーザー

- 大量のテキストファイル・マークダウンファイルをナレッジベースとして運用している個人
- プライバシーやセキュリティの観点からクラウドサービスを利用できない・したくないユーザー

## 3. 非機能要件

### 3.1 検索応答性（最重要）

- NFR-001: 全文検索の応答時間は1000ファイル規模で100ms以内を目標とする
- NFR-002: ベクトル検索の応答時間は1000ファイル規模（数万チャンク）で500ms以内を目標とする
- NFR-003: ハイブリッド検索の応答時間は1秒以内を目標とする
- NFR-004: LLM回答生成はストリーミング表示とし、最初のトークンが3秒以内に表示されることを目標とする
- NFR-005: 検索精度について、関連文書が上位10件以内に含まれる割合（Recall@10）を80%以上とする

### 3.2 プラットフォーム互換性

- NFR-010: Windows 10以降で動作する
- NFR-011: macOS 12 (Monterey) 以降で動作する
- NFR-012: ひとつのコードベースで両プラットフォームをサポートする

### 3.3 ローカル完結性

- NFR-020: 検索・回答生成は完全にローカルで動作する（ネットワーク不要）
- NFR-021: モデルダウンロード時のみネットワーク接続を要する

### 3.4 リソース効率

- NFR-030: アイドル時のCPU使用率は5%以内とする
- NFR-031: アイドル時のメモリ使用量は500MB以内とする
- NFR-032: インデックスサイズは元ファイル総サイズの2倍以内を目標とする

### 3.5 保守性

- NFR-040: TDD（テスト駆動開発）で開発する — 実践済み
- NFR-041: E2Eテストを自動化する（自動化不可の場合は手動テスト項目として管理する） — 実践済み
- NFR-042: ドキュメントを設計変更に追従して更新する — 実践済み

## 4. 技術制約（確定版）

| ID | 制約 | 採用技術・バージョン |
|---|---|---|
| TC-001 | Tauri v2をアプリケーションフレームワークとして使用する | tauri 2.10 |
| TC-002 | フロントエンドはReact + TypeScriptで実装する | React 19 + TypeScript 5.8 |
| TC-003 | ネイティブレイヤーはRustで実装する | Rust 1.94 |
| TC-004 | 全文検索エンジンはtantivyを使用する | tantivy 0.25 |
| TC-005 | 日本語形態素解析はlinderaを使用する | lindera 2 + IPAdic |
| TC-006 | ローカルLLMはllama.cppを使用し、単一バイナリに統合する | llama-cpp-2 0.1.140 |
| TC-007 | LLMモデルはGGUF形式を使用する | Qwen2.5系 |
| TC-008 | embeddingモデルはONNX Runtime + multilingual-e5-smallを使用する | ort 2.0.0-rc.12 |
| TC-009 | ベクトル検索はHNSWアルゴリズムを使用する | hnsw_rs 0.3 |

## 5. 想定規模

| 項目 | 想定値 |
|---|---|
| ナレッジベースのファイル数 | 約1,000ファイル |
| ファイルあたり平均サイズ | 5-50KB |
| ナレッジベース総サイズ | 5-50MB |
| チャンク数（ベクトル検索用） | 5,000-50,000 |
| embeddingモデルサイズ | 約470MB（ONNX） |
| LLMモデルサイズ | 491MB〜9GB（量子化レベルによる） |
